# Kiro 企业账号自助管理平台 · 产品文档

> Self-Service Kiro Account Provisioning with Feishu
>
> 面向使用 Kiro 企业版（IAM Identity Center 托管）的中国区客户：
> 把「建账号 → 加组 → 发密码邮件 → 订阅套餐 → 台账登记」的多步控制台操作，
> 收敛为一条「员工自助申请 → 飞书审批 → 全自动开通」的流水线。

---

## 1. 解决什么问题

企业客户落地 Kiro 时，账号管理是绕不开的运维负担：

| 痛点 | 现状 | 平台方案 |
|------|------|----------|
| 开通繁琐 | 管理员在 AWS 控制台手工：建 IDC 用户 → 加组 → 发密码邮件 → 订阅套餐，一人 4 步 | 审批通过后 4 步全自动执行，全步骤幂等 |
| 审批割裂 | 员工在 IM 里喊话，管理员开电脑进控制台 | 飞书卡片审批，手机上点「通过」即完成开通 |
| 台账缺失 | 谁有账号、谁在用、什么套餐，靠 Excel | DynamoDB 映射为唯一真相源 + 订阅实况实时对齐 |
| 用量黑盒 | 控制台只有订阅列表，看不到人维度用量 | 对接 Kiro 用量报表（Athena），按人聚合 credits/消息数 |
| 成本失控 | Overages 开了就是敞口，无人测算 | 超额上限展示 + 最坏敞口测算 + 应用内受控调高 |
| 存量混乱 | 平台上线前手工建的账号游离在外 | 一键扫描游离账号，智能建议归属，绑定即纳管 |

## 2. 用户视角：两个角色的完整旅程

### 员工（自助）

1. 飞书扫码登录 Web 门户（OAuth，无需另设密码）
2. 「申请账号」：系统按姓名拼音自动推荐可用用户名，选分组和套餐提交
   - 提交即校验：用户名占用（平台台账 + IDC 实查双层）、邮箱唯一——错误当场提示，不浪费审批
3. 等待审批（飞书会收到结果通知），通过后查收密码设置邮件，登录 Kiro 开始使用
4. 后续自助：查看名下所有账号的套餐/订阅状态（实时）、发起升级、跟踪申请记录（含失败原因）

### 管理员（治理）

飞书收到审批卡片，**手机上直接批**；或进 Web 审批后台，五个视图：

| Tab | 能力 |
|-----|------|
| 申请审批 | 待审批列表，通过/拒绝（与飞书卡片同源防重，谁先点谁生效） |
| 账号总览 | 指标卡（用户数/账号数/总 credits/超额上限）+ Top 用户图表 + Tier 分布 + 按人聚合表（一人多账号合并，可展开下钻） |
| 分组管理 | 直接新建 IAM Identity Center Group，申请表单下拉实时生效 |
| 存量导入 | 扫描 IDC 游离账号 → 邮箱/拼音双级归属建议 → 确认绑定纳管 |
| （卡片内）| Overages 超额上限应用内调高（护栏 + 审计） |

## 3. 核心功能清单

### 3.1 自助申请与审批开通

- 飞书 OAuth 登录 + Web 表单申请；姓名拼音自动推荐用户名
- 审批双入口：飞书交互卡片（免公网，WS 长连接收回调）/ Web 后台，同一状态机防重
- 通过后自动执行 4 步：IDC 建用户 → 加组 → 密码设置邮件 → Kiro 订阅，分步错误定位，全步骤幂等
- 4 档套餐全支持：Pro ($20) / Pro+ ($40) / Pro Max ($100) / Power ($200)
- 申请、审批、执行结果全程飞书通知闭环；失败原因同时展示在 Web 申请记录

### 3.2 账号台账与订阅实况

- DynamoDB 映射表为唯一真相源：以 IDC UserId 为锚点（不随改名漂移），支持一人多账号（主/副自动判定）
- **订阅实况对齐**：套餐/状态每次展示都与 AWS 订阅数据实时比对——管理员在控制台直接退订/改套餐，平台界面同步反映，不会出现台账与现实脱节
- 升级智能分流：活跃订阅走变更；已取消订阅自动改走重新订阅，并对「账期未结占位」给出明确提示

### 3.3 用量看板（对接 Kiro 用量报表）

- 数据源：Kiro Console 的 User Activity Report（CSV → Glue → Athena）
- 按人聚合：credits、消息数、会话数、活跃天数、最后活跃时间；一人多账号自动汇总
- 纯 SVG 自绘图表（Top 用户柱状图、Tier 分布环形图），无重型依赖
- 副账号回收决策支持：筛选低用量副账号，生成回收依据

### 3.4 成本治理（Overages）

- 展示 AWS Service Quotas 中「Maximum allowed overage per Kiro profile」当前值
- **最坏月超额敞口测算**：账号数 × 上限，一眼看到风险面
- 应用内发起调高：三道护栏（必须大于当前值 / 单次最多 2 倍 / 有待审批请求时禁止重复提交）+ 二次确认 + 落审计表（谁、何时、从多少到多少）
- 明确提示单向性：上限只可调高，调低需 AWS Support case

### 3.5 存量账号导入

- 一键扫描：IDC 全量 diff 平台台账，找出游离账号（平台上线前手工建的）
- 双级归属建议：邮箱精确匹配（高可信，绿标）▶ 用户名拼音基名匹配（需人工核对，橙标）
- 确认绑定后立即纳管：出现在员工「我的账号」、套餐/状态走实况、用量自动关联

### 3.6 分组管理

- 管理端直接创建 IAM Identity Center Group（分组即 IDC 组，开通时账号真实加组）
- 申请表单分组下拉从 IDC 实时拉取，换环境零改码

## 4. 安全基线

| 项 | 做法 |
|----|------|
| 凭证 | 全程 IAM Role（EC2 Instance Profile / ECS Task Role / EKS IRSA），代码层无任何 AK/SK 读取路径 |
| 权限 | 最小权限 IAM Policy 随仓库提供（`infra/iam-policy.json`） |
| 网络 | 飞书回调走 WS 长连接，**无需公网入站**；Web 端可放内网或加隧道 |
| 认证 | 飞书 OAuth + JWT；管理员白名单（open_id 列表） |
| 防重 | 审批状态机条件更新，多管理员并发点击只有一人生效 |
| 审计 | 申请/审批/执行/超额调整全量落 DynamoDB requests 表 |
| 配置 | 密钥类配置仅 `.env`（gitignore），支持 SSM Parameter Store SecureString 分发 |

## 5. 技术架构

```
员工/管理员
   │ 飞书 OAuth / 卡片审批（WS 长连接，免公网）
   ▼
FastAPI (Python 3.12) ── React + AntD (TS) 同源托管
   │
   ├─ IAM Identity Center   建用户 / 加组 / 组管理 / 存量扫描
   ├─ Kiro 订阅 API          订阅 / 升级 / 取消 / 实况查询
   ├─ Service Quotas        Overages 上限查询 / 受控调高
   ├─ Athena + Glue         用量报表聚合（TTL 缓存，降级安全）
   └─ DynamoDB × 2          账号映射（真相源）+ 申请审计
```

- 架构图：[docs/architecture-aws.png](./architecture-aws.png)
- 所有外部依赖都做了**降级设计**：用量/配额/实况任一数据源不可用时，界面显示占位符，核心开通链路不受影响
- 单 EC2 t3.small 即可承载数百人规模；无状态设计，可平迁 ECS/EKS

## 7. 边界与已知限制

如实说明，避免落地时才发现：

- **Kiro 订阅管理无公开 SDK**：订阅相关调用采用与控制台一致的服务接口手动签名实现，AWS 侧接口变更可能需要适配（仓库内已隔离在单一模块，便于维护）
- **Overages 上限只可调高**：AWS Service Quotas 的机制限制，调低需开 Support case；平台已做护栏与提示
- **已取消订阅当月占位**：取消后账期保留至月底，期间重新订阅会被服务端拒绝，平台会给出明确提示
- **存量导入的绑定目标**：仅覆盖在平台登录/申请过的飞书用户（平台默认不申请通讯录权限）；如需全员名单可增配飞书通讯录只读权限
- 定位为 **sample code**（MIT-0），生产部署前请按贵司安全合规要求评审
